# Context
`my-tank/myth-tank.js` 现在已经变成“规则越加越碎、修一处就牵一片”的状态：为了防双弹加的逻辑，会影响普通躲避；为了防卡墙加的脱困，会影响追星和站位；为了防某个局面写的例外，会把整套策略改怂。用户现在要的不是继续补丁式修修补补，而是把现有规则树重构成一套**评分系统 + 清晰分层的坦克代码架构**，并且尽量把所有现有行为都纳入新体系，而不是只挑几个好做的先替换。

README 已经给出方向：动作要候选化、统一打分；必死情况要硬约束直接禁止；还要有“勇敢基线”，避免坦克因为过度防守而变得不敢抢星、不敢压制、不敢开火。

# Recommended approach
1. **先冻结现有行为，建立可对照的基线入口**
   - 保留 `my-tank/myth-tank.js` 作为当前可运行入口，但把它定位成“决策外壳”，而不是继续承载所有规则细节。
   - 先不要删除老逻辑；新架构必须先能和旧逻辑并行对照、逐步替换，才能避免把坦克一次性改坏。
   - 增加可解释的决策输出（动作类型、分数、被否决原因），让后续调权重时能直接看出为什么选了这个动作。

2. **把决策重构成统一管线，不再靠大段 if/return 链堆规则**
   - 新的 `onIdle` 只负责编排：
     1. 采集本帧状态
     2. 更新跨帧记忆
     3. 先过硬生存闸门
     4. 生成所有候选动作
     5. 用评分器排序
     6. 选择最优合法动作并执行
   - 这里的关键不是把规则“少写几条”，而是把“什么时候是判断、什么时候是提案、什么时候是裁决”分清楚。

3. **把全局规则拆成三层：生存层、收益层、计划层**
   - **生存层**：只判断“这一步会不会立刻送命”
     - 子弹躲避
     - 紧急传送
     - 两步脱困
     - 绝境横移
     - 防瞄
     - 近距对射规避
     - 双弹带/炮线/死亡落点/死胡同等硬禁区
   - **收益层**：只判断“这一步值不值得做”
     - 抢星、控星线、守线开火
     - 草丛伏击
     - 敌炮管空窗期反击
     - 传送抢星
     - 传送刺杀
     - 巡逻、绕位、压制
   - **计划层**：只判断“跨帧意图怎么保持稳定”
     - 巡逻目标粘性
     - 短期意图缓存
     - 蹲草等待
     - 之字规避
     - 敌人逃跑趋势记忆
     - 卡住/震荡恢复
   - 这三层必须都保留；不是只先做一部分，而是从一开始就按这个边界搭完整架构。

4. **建立统一的动作模型，把所有现有行为都变成“候选动作”**
   - 每个候选动作用同一种结构表达，例如：
     - `type`
     - `step` / `target`
     - `preconditions`
     - `risk`
     - `reward`
     - `stability`
     - `horizon`
     - `tags`
     - `reason`
   - 评分器不要直接理解“某个 if 分支”，而是只理解这些候选动作。
   - 这样以后新增行为时，只是新增一个候选提案，不再往主决策链里塞一段新分支。

5. **把 README 里的原则完整落成评分维度，而不是只做“能打分”这件事**
   - **收益维度**：
     - 抢星收益
     - 星线控制收益
     - 压制收益
     - 击杀收益
     - 侧翼/背后收益
     - 终局锁分收益
     - 破墙开路收益
     - 走位占位收益
   - **风险维度**：
     - 当前/未来几帧子弹轨道
     - 双弹覆盖带
     - 敌人炮口已对准
     - 近距同线换血风险
     - 死胡同/封锁走廊/贴墙窄兜
     - 传送落地即死
     - 无意义让位导致丢分
   - **稳定性维度**：
     - 是否能保持 2~4 步连续意图
     - 是否会在等价候选间抖动
     - 是否会因为局部最优反复横跳
   - **勇敢基线**：
     - 只要不是硬危险，就不能因为分数压得太重而变成“安全但不出手”的坦克。
   - **硬约束**：
     - 必死、必落弹道、必进双弹带、必入死亡落点的动作直接丢弃，不参与打分。

6. **把当前规则函数系统化成“提案器”，并保留对现有行为的完整覆盖**
   - 现有函数先不删，先分组改成提案来源：
     - **生存提案**：`findBulletDodge`、`findEscapeTeleport`、`findTwoStepEscape`、`findDesperateDodge`、`findAimDodge`、`findLineDuelDodge`
     - **攻击提案**：`findEnemyBulletOpenShot`、`findGuardLineShot`、`findBushLineShot`、`findAssassinationPlan`
     - **目标提案**：`findStarTeleport`、`shouldChaseStar`、`findContestedStarGuard`
     - **移动提案**：`virtualPatrolTarget`、`nextStepToStandoff`、`nextStepToSafeBush`、`escapeDoubleLaneBand`、`escapeAmbushLine`、`diagonalEvadeStep`、`safestNonDeadEndStep`、`bestSafeNeighbor`、`breakStuckStep`、`findDigDirection`
   - 这些函数不再是“谁先命中就谁赢”的最终裁判，而是候选动作生成器。
   - 最终决策由评分器统一裁决，避免规则之间互相抢优先级。

7. **把跨帧状态收口成“状态存储层”，不要继续让业务状态散落在主流程里**
   - 保留真正有价值的跨帧记忆：
     - `lastEnemyPos`
     - `lastEnemySeenFrame`
     - `stuckFrames`
     - `lastMyPos` / `lastMyPos2`
     - `enemyFleeFrames`
     - `patrolTarget`
     - `shortIntent`
     - `pendingAssassin` / `assassinBanned`（如果刺杀策略仍保留）
   - 但要把它们从“规则混杂状态”里抽出来，变成统一的状态对象或状态层接口。
   - 太局部的启发式记忆（例如只给之字规避服务的轴偏好）尽量收回到局部评分或局部执行器里，避免污染全局决策。

8. **完整覆盖当前已知行为族，避免“新架构只照顾一部分”**
   - 这个重构不是只为了抢星或巡逻，而是要把所有当前战术都纳入统一系统：
     - **生存类**：子弹躲避、紧急传送、两步脱困、绝境横移、防瞄、近距对射规避、双弹带脱离、冻结/过载的特殊死区
     - **攻击类**：空窗期反击、守线开火、草丛伏击、传送刺杀、对盾流的开火策略、对双弹流的保守策略
     - **目标类**：星星抢夺、传送抢星、终局抢星、星线争夺、对手跑路时的追星放宽
     - **机动类**：巡逻目标、绕位、之字逃离、靠墙破局、安全徘徊、最佳邻格
     - **记忆类**：敌人最后可见位置、逃跑倾向、卡住状态、短期意图连续性
   - 新架构必须能覆盖这些家族，否则只是把规则换了个壳，后面还是会继续坏味道。

9. **迁移顺序采用“先搭完整框架，再分批搬入”，但不是只做少量行为**
   - 先建立新的决策引擎、评分器、状态层和动作提案接口。
   - 再把所有现有行为按家族映射进去，确保每个家族都有对应提案和评分权重。
   - 最后才逐步删掉旧的硬返回逻辑，改成由统一引擎裁决。
   - 这样不是“只先做四类容易的行为”，而是“先把整套系统搭好，再分批把全部行为迁进去”。

10. **测试要按“行为族 + 冲突场景 + 稳定性”三条线重建**
    - 保留 `_scenario_test.js` 里的现有回归，作为旧行为的底座。
    - 追加三类新的测试轴：
      1. **生存冲突**：子弹轨道、双弹带、紧急传送、死亡落点、近距同线换血。
      2. **收益冲突**：高收益动作与硬危险冲突时，必须拒绝送命动作；高收益动作与低风险动作冲突时，必须选收益更高的。
      3. **计划稳定性**：巡逻、蹲草、之字、追星、绕位要能跨帧保持意图，不能每帧抖动。
    - 重点不只是测“会不会死”，而是测“候选动作之间的裁决是否符合评分系统”。

# Critical files
- `my-tank/myth-tank.js`：当前决策入口，先拆成薄外壳，再迁移到评分引擎
- `my-tank/_scenario_test.js`：回归基线和新评分场景都放这里验证
- `README.md`：评分系统设计原则的来源，作为 reward / penalty / hard-gate 的规范依据
- `my-tank/decision-engine.js`（建议新增）：统一调度硬闸门、候选提案、评分与执行
- `my-tank/scoring.js`（建议新增）：维护动作评分、权重、解释输出、勇敢基线修正
- `my-tank/action-proposals.js`（建议新增）：把当前规则函数逐步改成候选动作提案器
- `my-tank/state-store.js`（建议新增）：收口跨帧状态、计划缓存与对局生命周期
- `my-tank/decision-trace.js`（建议新增，可选）：输出每个候选动作的分数与淘汰原因，便于调参

# Verification
1. 先跑 `node my-tank/_scenario_test.js`，确认重构前后的回归基线保持一致。
2. 为每个行为族补一组评分冲突测试，至少覆盖：
   - 生存 vs 收益
   - 收益 vs 硬约束
   - 计划稳定性 vs 临时诱惑
3. 在新引擎里开启决策 trace，检查每次失败时到底是哪个维度压低了分数或触发了硬约束。
4. 逐步迁移时，优先验证每个行为族的“最坏情况”而不是只看理想局。
5. 最后再做真实局/模拟验证，确认新评分系统没有把坦克改成保守、抖动、或者只会躲不敢打的版本。
